You are an expert in categorizing cloud infrastructure incidents reported by major cloud providers (Azure, AWS, GCP). Analyze the incident report step by step to extract structured information. Follow the reasoning steps:

1. Identify the vendor (Azure, AWS, or GCP) and the specific service name affected. Use the official product name as it appears in the report.

2. From {service_category_lst}, select one most relevant service category.
   - COMPUTE: virtual machines, containers, serverless functions, batch processing
   - STORAGE: object storage, file storage, disk storage, backup services
   - NETWORK: load balancers, CDN, DNS, VPN, peering, connectivity services
   - SECURITY: identity, authentication, key management, certificates, WAF
   - AI: machine learning, AI/LLM APIs, cognitive services, vision, speech
   - MANAGEMENT: monitoring, logging, alerting, deployment, orchestration tools
   - ANALYTICS: data warehouses, streaming pipelines, BI tools, data processing
   - DATABASE: relational databases, NoSQL, caching, in-memory stores
   - OTHERS: services that do not clearly fit the above categories
   - UNKNOWN: service category cannot be determined from the report

3. Extract the relevant sentence(s) directly from the report that describe user-visible symptoms. Then, from {user_symp_lst}, select one or more categories that best match.
   The definition for user_symptom_category: {user_symp_instruction}
   - Be conservative: only label what is explicitly stated. Prefer a single label unless the report clearly describes two distinct symptom types.

4. Extract the relevant sentence(s) directly from the report that explain the root cause. Then, from {root_cause_lst}, select exactly one category that best matches.
   The definition for root_cause_category: {root_cause_instruction}
   - Distinguish DEPLOY (bad rollout or software bug) from CONFIG (pre-existing or manually applied misconfiguration).
   - Distinguish OVERLOAD (demand exceeded capacity) from MAINTAIN (planned maintenance with unexpected side effects).
   - If a third-party dependency caused the issue, use EXTERNAL.

5. Identify the start time and end time in the format YYYYMMDDThh:mm:ss (e.g., 20230412T14:00:00). Use the time exactly as stated in the report — do not convert to UTC. If only a date is given, append T00:00:00. If only a time is given without a date, use the format hh:mm (e.g., 14:00). Use "UNKNOWN" if neither date nor time can be determined.
   - For AWS timeline-style reports: each paragraph begins with a timestamp of when that update was posted — ignore these post timestamps. The actual incident window appears as a "Between X and Y" sentence inside the final resolved update. Use those X and Y values as start_time and end_time.
   - If no "Between X and Y" sentence exists but the first update timestamp marks the beginning of the incident, use that as start_time and the last update timestamp as end_time.
   - For GCP reports: start_time is stated as "beginning at <time>" and end_time as "resolved ... as of <time>" inside the text fields.

Finally, return the extracted information in a JSON object with the keys:
vendor, service_name, service_category, start_time, end_time, user_symptom, user_symptom_category, root_cause, root_cause_category.

Use "UNKNOWN" for any field that cannot be determined. Do not copy boilerplate apology or survey language into symptom or root cause fields.
